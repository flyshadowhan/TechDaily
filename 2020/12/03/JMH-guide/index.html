Java基准测试工具套件JMH入门教程

发表于 2020-12-04 分类于 JMH 阅读次数：
本文字数： 7.4k 阅读时长 ≈ 7 分钟
JMH是Java平台的基准测试工具套件,是Java Microbenchmark Harness(微基准测试)框架的缩写,由Hotspot JVM团队开发.本文是JMH的入门教程.

背景

如何评价Java代码的执行效率的高低?

  • 开发环境,将执行代码段放入for循环中,对循环进行计时,结束后计算每次执行时间

这种方式多为定性的结果,并且对于测试环境依赖较大,不具备可重复性.是否存在一种可以定量并且对测试环境不依赖的测试方法呢?当然,性能测试中有一类的测试称为基准测试(Benchmark) .

基准测试是通过设计科学的测试方法、测试工具和测试系统,实现对一类测试对象的某项性能指标进行定量的和可对比的测试.可测量、可重复、可对比是基准测试的三大原则.

针对PC硬件和整体系统的基准测试软件早已发展多年,如早期的SPEC,现在PCMARK/3DMARK,还发展出针对移动端的版本.

JMH介绍

JMH(Java Microbenchmark Harness)是用于代码微基准测试的工具套件,主要是基于方法层面的基准测试,精度可以达到纳秒级,由Hotspot JVM团队开发.JMH于2013年首次发布.

JMH支持完整的基准测试过程,包括预热、运行、统计和报告等,支持除Java外的其他JVM语言.更重要的是,它针对Hotspot JVM提供了各种特性,来保证基准测试的正确性.

不少资料提到JMH集成至JDK12中,应该是个误解.微基准套件只是作为特性添加到了JDK的源码库中,并未进入发行版.How to use Java 12’s Microbenchmark Suite?

API的说明

@BenchmarkMode 基准测试的运行模式

  • Mode.Throughput - 吞吐量,单位时间内执行多少次
  • Mode.AverageTime - 单次调用平均时间
  • Mode.SampleTime - 随机取样,最后输出取样结果的分布,输出结果与jmeter聚合报告中的n% line类似
  • Mode.SingleShotTime - 只运行一次,用于测试冷启动时的性能
  • All - 上述所有模式

@Warmup 预热

JVM会尝试将某个调用多次的函数对应的字节码转换成机器码,从而提高执行速度.这个过程被称为JIT(just-in-time compilation).为了让基准测试结果更加接近实际情况,就需要通过多轮预热来触发JIT,提升执行效率.

  • iterations - 预热的轮数
  • time - 每次预热的时间
  • timeUnit - 时间单位,默认是秒

@Measurement 基准测试相关参数

说明同@Warmup

@OutputTimeUnit 测量的时间精度

测量的时间精度,一般用毫秒或微秒,最高支持到纳秒.

@Fork 测量迭代数

  • value - 执行测量迭代的次数

@Threads 执行当前测试的线程数

  • value - 执行线程数,默认值为Runtime.getRuntime().availableProcessors()

@State 共享变量范围

  • Scope.Benchmark - 所有线程间共享变量
  • Scope.Group - 同一个组内的线程共享变量
  • Scope.Thread - 每个线程单独一份变量,线程之间相互隔离

@Param 测试入参

对应测试用例里面的变量n,控制字符串相加的次数(1次/10次/100次)

@Setup 初始化方法
测试前相关准备操作,类似Junit @Before

@TearDown 结束方法
测试结束相关操作,类似Junit @After

@Benchmark 基准测试方法
需要进行基准测试的方法上标记该注解,方法的访问级别必须为public

报告的说明

测试报告会显示比较多的信息,一般关注最后的评分结果.这里报告中相关字段含义如下.

  • (n)
    对应@Param取值
  • Mode
    基准测试的运行模式
  • Cnt
    总的执行次数,Cnt = Fork * Measurement.iterations
  • Score
    执行测试得分
  • Error
    非错误的含义,翻译为置信区间更合适,相当于得分的上下限.
    参考报告中的内容: 2.675 ± 0.269 = [2.406, 2.944]
  • Units
    得分的单位
